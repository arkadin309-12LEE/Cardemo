# 카데모(CARDEMO) Phase 2 상세 작업계획서 (초심자용)

## 1. Phase 2 목표

이 단계의 목표는 **카드를 체계적으로 그룹화하는 덱(DECK) 기능을 완성**하는 것입니다. 낱장의 카드들을 의미 있는 묶음으로 만들고, 덱 자체를 꾸미는 재미를 더하여 앱의 핵심적인 정보 관리 능력을 갖추게 됩니다.

---

## **Sprint 5 (5주차): 핵심 덱 기능 구현**

> **이번 주 목표**: 사용자가 덱을 만들고, 카드를 그 안에 넣고 빼는 가장 기본적인 덱 관리 기능을 구현합니다.

- [ ] **덱(Deck) 모델 클래스 및 DB 연동**
    - **(힌트)**: 기획서의 `decks` 테이블 구조에 맞춰 `Deck` 클래스를 만드세요. `Card` 모델과 마찬가지로 `toMap()`, `fromMap()` 메소드를 만들어 DB와 소통할 준비를 합니다.
    - **(예상 소요시간)**: 4시간

- [ ] **덱 리스트 화면 UI 구현**
    - **(힌트)**: 카드 리스트 화면과 유사하게, `ListView.builder`를 사용하여 덱 목록을 보여줄 화면을 만듭니다. 우측 하단에 `FloatingActionButton`을 추가하여 새 덱을 만드는 기능으로 연결합니다.
    - **(예상 소요시간)**: 5시간

- [ ] **덱 생성 기능 구현**
    - **(힌트)**: 새 덱 생성 버튼을 누르면, 덱의 이름과 설명을 입력하는 대화상자(AlertDialog) 또는 별도의 페이지를 띄웁니다. 입력된 정보로 `Deck` 객체를 만들어 DB에 `INSERT`합니다.
    - **(예상 소요시간)**: 5시간

- [ ] **덱에 카드 추가/제외 기능 구현**
    - **(힌트)**: 카드 리스트에서 여러 카드를 선택할 수 있는 모드를 만듭니다. 선택된 카드들의 `deck_id` 필드를 특정 덱의 ID로 `UPDATE`하면 됩니다. 덱에서 카드를 제외하는 것은 `deck_id`를 `null`로 업데이트하는 것과 같습니다.
    - **(예상 소요시간)**: 8시간

---

## **Sprint 6 (6주차): 덱 관리 및 조회**

> **이번 주 목표**: 사용자가 덱 목록을 보고, 특정 덱을 선택했을 때 그 안의 카드들만 모아볼 수 있는 기능을 완성합니다.

- [ ] **덱 리스트 기능 구현**
    - **(힌트)**: DB의 `decks` 테이블에서 모든 덱 정보를 `SELECT`하여 덱 리스트 화면에 표시합니다.
    - **(예상 소요시간)**: 4시간

- [ ] **덱 내부 카드 조회 기능**
    - **(힌트)**: 덱 리스트에서 특정 덱을 선택하면, 해당 덱의 `id`를 카드 리스트 화면으로 전달합니다. 카드 리스트 화면은 전달받은 `deck_id`를 이용해 `SELECT * FROM cards WHERE deck_id = ?` 쿼리로 카드들을 필터링하여 보여줍니다.
    - **(예상 소요시간)**: 8시간

- [ ] **덱 정보 수정 및 삭제 기능**
    - **(힌트)**: 덱 리스트에서 덱을 길게 누르면 나오는 메뉴를 통해 덱의 이름과 설명을 수정(`UPDATE`)하거나, 덱 자체를 삭제(`DELETE`)하는 기능을 구현합니다. 덱 삭제 시, 안에 있던 카드들의 처리 방법(같이 삭제할지, 낱장 카드로 둘지)을 결정해야 합니다. (기획: 낱장 카드로 변경)
    - **(예상 소요시간)**: 8시간

---

## **Sprint 7 (7주차): 덱 커스터마이징 (이미지)**

> **이번 주 목표**: 밋밋한 덱 목록에 생기를 불어넣습니다. 각 덱을 대표하는 이미지를 설정할 수 있게 합니다.

- [ ] **덱 커버 이미지 설정 기능**
    - **(힌트)**: 덱 편집 화면에 ‘커버 이미지 변경’ 버튼을 추가합니다. 3주차에 사용했던 `image_picker` 패키지를 재활용하여 이미지를 선택하고, 해당 파일 경로를 `decks` 테이블의 `cover_image` 컬럼에 `UPDATE`합니다.
    - **(예상 소요시간)**: 6시간

- [ ] **덱 리스트에 커버 이미지 표시**
    - **(힌트)**: 덱 리스트의 각 항목(아이템) UI를 수정합니다. `Image.file()` 위젯을 사용하여 저장된 `cover_image` 경로의 이미지를 표시합니다. 이미지가 없다면 기본 아이콘이나 색상 배경을 보여주도록 처리합니다.
    - **(예상 소요시간)**: 6시간

- [ ] **덱 보기 화면 상단에 커버 이미지 표시**
    - **(힌트)**: 덱 내부 카드 리스트 화면의 상단(AppBar 또는 배경)에도 선택된 커버 이미지를 표시하여, 사용자가 어떤 덱을 보고 있는지 시각적으로 인지할 수 있게 합니다.
    - **(예상 소요시간)**: 6시간

---

## **Sprint 8 (8주차): 덱 타이틀 커스터마이징**

> **이번 주 목표**: 덱 꾸미기의 핵심 기능입니다. 사용자가 덱 커버 이미지 위에 제목을 자유롭게 배치하고 꾸밀 수 있게 합니다.

- [ ] **타이틀 편집 UI 구현**
    - **(힌트)**: 이 기능은 복잡합니다. `matrix_gesture_detector` 또는 `interactive_viewer` 같은 패키지를 사용하면 드래그, 스케일, 회전 제스처를 쉽게 감지할 수 있습니다. 덱 커버 이미지 위에 `TextField` 대신 `Text` 위젯을 올리고, 이 위젯을 제스처로 조작할 수 있도록 구현합니다.
    - **(예상 소요시간)**: 10시간

- [ ] **타이틀 위치, 크기, 회전 값 저장 및 적용**
    - **(힌트)**: 제스처를 통해 얻어진 변환 정보(Transformation Matrix)를 DB에 저장해야 합니다. 매트릭스 값을 문자열로 저장하거나, 위치(x, y), 크기(scale), 회전(rotation) 값을 각각 별도의 컬럼에 저장하는 방법을 사용할 수 있습니다. 덱을 불러올 때 이 값을 `Transform` 위젯에 적용하여 타이틀을 표시합니다.
    - **(예상 소요시간)**: 8시간

- [ ] **타이틀 스타일(폰트, 색상) 변경 기능**
    - **(힌트)**: 타이틀 편집 모드에서 하단에 색상 선택기, 폰트 리스트 등을 표시하여 사용자가 타이틀의 스타일을 변경하고, 이 정보를 DB에 저장/적용하는 기능을 구현합니다.
    - **(예상 소요시간)**: 4시간

---

## **Sprint 9 (9주차): 고급 검색 및 정렬**

> **이번 주 목표**: 카드와 덱이 많아졌을 때를 대비하여, 원하는 정보를 더 빠르고 쉽게 찾을 수 있도록 검색과 정렬 기능을 강화합니다.

- [ ] **덱 포함 전체 검색 기능**
    - **(힌트)**: 기존 카드 검색 기능에 ‘덱 포함 검색’ 토글 스위치를 추가합니다. 이 옵션이 켜지면, 검색 범위에서 낱장 카드뿐만 아니라 모든 덱에 포함된 카드들까지 검색하도록 SQL 쿼리를 수정합니다.
    - **(예상 소요시간)**: 6시간

- [ ] **카드/덱 리스트 정렬 기능**
    - **(힌트)**: 리스트 화면 상단에 ‘정렬’ 아이콘 버튼을 추가하고, 터치 시 ‘최신순’, ‘이름순’ 등 정렬 옵션을 보여주는 메뉴(PopupMenuButton)를 띄웁니다. 선택된 옵션에 따라 `ORDER BY` 절이 포함된 SQL 쿼리를 실행하여 리스트를 새로고침합니다.
    - **(예상 소요시간)**: 8시간

- [ ] **검색 및 정렬 상태 유지**
    - **(힌트)**: 사용자가 설정한 정렬 순서나 검색 키워드를 앱을 껐다 켜도 유지되도록 `shared_preferences` 같은 간단한 저장소에 저장하고, 화면이 로드될 때 이 값을 불러와 적용합니다.
    - **(예상 소요시간)**: 4시간

---

## **Sprint 10 (10주차): 2단계 통합 및 안정화**

> **이번 주 목표**: 지금까지 만든 모든 카드와 덱 관련 기능들이 서로 문제없이 잘 동작하는지 종합적으로 테스트하고 다듬습니다.

- [ ] **Phase 1, 2 기능 통합 테스트**
    - **(힌트)**: 카드를 만들고, 이미지를 넣고, 덱에 추가한 뒤, 덱 커버와 타이틀을 꾸미고, 그 덱 안에서 카드를 검색하는 등 사용자가 할 수 있는 모든 행동을 처음부터 끝까지 테스트합니다.
    - **(예상 소요시간)**: 8시간

- [ ] **테스트 코드 보강**
    - **(힌트)**: 덱 생성, 카드 추가/제외 등 2단계에서 추가된 핵심 로직들에 대한 단위 테스트(Unit Test) 코드를 작성하여 코드의 안정성을 높입니다.
    - **(예상 소요시간)**: 6시간

- [ ] **2단계 버그 수정 및 안정화**
    - **(힌트)**: 통합 테스트 과정에서 발견된 모든 버그를 수정합니다. 특히 덱에서 카드를 제외하거나 덱을 삭제했을 때 데이터가 꼬이는 부분이 없는지 집중적으로 확인합니다.
    - **(예상 소요시간)**: 8시간
